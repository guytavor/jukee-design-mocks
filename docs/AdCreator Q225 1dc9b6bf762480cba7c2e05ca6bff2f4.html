AdCreator Q225

Jukee Audio‑Ad Creator – vNext Spec

Note: this document describes the first unified creator release. The Tweak panel will intentionally start lightweight and will be expanded in later versions—adding deeper timeline editing, multiple SFX placement, length changes, and other controls—once real‑world customer feedback tells us what matters most.

1. Goal

  • Merge AdTune Creator and Announcement Creator into one streamlined Gen‑AI‑first flow.
  • Provide instant draft generation and lightweight post‑generation tweaks without exposing a full DAW.

2. Gen‑AI Prompt UI

2.1 Layout

ZonePurposeKey UX Notes
A. Prompt Text AreaFree‑text multiline, 70 % widthAutofocus; placeholder example; live validation hints
B. Smart‑Chip RailClickable chips (tone, length, etc.) 30 % width (scrolls on mobile)Chips append/update key:value tokens in prompt
C. Action BarGenerate (primary), Reset, Sample PromptsSticky bottom; disable Generate until prompt ≠ empty
D. Output CarouselTwo auto‑mixed previews + RegenerateHidden until first run; A/B switch; waveform thumb

2.2 Chip Groups & Tokens (MVP)

  • Tone / Style – hard_sell, soft_sell, story_teller, friendly_informative, urgent_flash_sale
  • Duration – 6s, 12s, 20s, 30s
  • Music Style – Funk, Synth‑pop, Acoustic_Chill, Orchestral, no_music
  • Voice Preference – male, female, child, bilingual, celebrity_like , brand_voice if applicable
  • Chime / Sound‑Logo – brand, no_chime, retailer_jingle (only appear if tenant_assets include the chime and the sound_logo)
  • SFX – modal picker with search & audio preview; selected SFX become sfx:<id> tokens
  • Call‑to‑Action Tags – quick‑insert phrases (not tokens) e.g. Buy Now, Today Only
  • Template tags - $price, $shelf-location

2.3 Prompt → JSON Rules

  • One token per chip group (chip wins over conflicting free‑text).
  • length sets target_seconds.
  • chime boolean selects fixed asset.
  • sfx_events initially empty; populated later via Tweak.

2.4 Generation Flow

  1. Generate → spinner "Cooking your ad…" (≈5 s)
  1. Backend returns JSON + 2 MP3 URLs
  1. Output Carousel appears; first preview auto‑plays (muted)
  1. Buttons: Use this, Tweak, Play

2.5 JSON file for communication between Gateway and UI

2.5.1. JSON Render_spec example:

{
  "script_text": "Buy now before it's gone!",
  "voice_id": "alex",
  "music_bed_id": "funk_v2",
  "narr_offset_ms": 800,
  "music_gain_db": -2.5,
  "include_chime": true,
  "include_logo": false,
  "length": "12s",
  "tone": "hard_sell",
  "sfx_events": [
    { "asset_id": "cash_register", "offset_ms": 1000, "gain_db": -1 },
    { "asset_id": "whoosh", "offset_ms": 2000, "gain_db": -2 }
  ]
}
  • script_text: Final voiceover narration
  • voice_id: Selected voice actor ID
  • music_bed_id: Background track asset ID
  • narr_offset_ms: Delay before narration starts
  • music_gain_db: Volume control for music bed
  • include_chime / include_logo: Booleans for audio branding elements
  • length: Intended duration (optional override to template logic)
  • tone: Used by template selector (if multiple timing/mix configs exist)
  • sfx_events: List of extra audio snippets, each with timing + volume

3. “Tweak” Mode v0

3.1 Scope (fast fixes only)

ControlJSON FieldReason
Narration text editscript_textTypos, legal copy changes
Voice‑actor selectorvoice_idGender / persona switch
Music‑bed pickermusic_bed_idVibe adjustment
Narration offset slider (0–5 s)narr_offset_msSync VO with bed
Background‑music volumemusic_gain_dbBalance VO vs bed
Chime toggleinclude_chimeRetailer preference
Add SFX (modal reuse)sfx_events[{asset_id, offset_ms, gain_db}]Extra flair

3.2 SFX in Tweak

  • Add SFX → modal <search/preview> → on select, chip appears in list.
  • Default offsets stagger 1 s apart after VO start; slider (0–5 s) per SFX overrides.
  • Global SFX limit = 3; warn & block beyond.
  • Undo restores previous JSON.

5. Chip Selection Sync Logic

  • User‑clicking a chip inserts or updates a key:value token into the prompt.
  • Every key:value token is parsed from the prompt and stored in React state (tokens).
  • UI chips render as active if their key:value matches state.
  • If user edits the prompt and deletes or rewrites a token, chip states auto‑update.
  • All input flows are reversible: prompt → tokens → chips → prompt.

6. History Management

  • Every generated ad version is saved to history record in the JSON
  • Each entry includes: timestamp, status (draft/saved/published), full JSON, and preview URL.
  • History panel (right-side drawer) lists all previous versions chronologically.
  • Each entry shows: tone, music, duration, preview text, status, and actions: Use, Delete.
  • Drafts expire after 7 days unless promoted to saved.
  • “Use” button loads that JSON into editor and replaces the current working draft.

8. Gateway API for Unified Ad Creator

The following Gateway APIs support the new unified AdCreator vNext experience.

They provide endpoints for generation, tweaking, session management, asset browsing, and finalization — enabling a fast and flexible creative workflow.

8.1. Editor Versioning

Every session and saved ad is associated with a creator_version.

  • Current: "vNext"
  • Older ads may have creator_version unset (legacy Announcement or AdTune).

Backend must store creator_version: vNext in all new or migrated documents.

Frontend checks this to decide which editor UI to load.

8.3. Generation

Route:

POST /audio_ad_creator/generate

Description:

Creates two initial draft ad previews from the prompt and selected tokens.

Request Body: Render_spec JSON

Response:

{
  "generated_previews": [
    {
      "version_id": "v1",
      "preview_url": "https://...",
      "render_spec": { ... }
    },
    {
      "version_id": "v2",
      "preview_url": "https://...",
      "render_spec": { ... }
    }
  ]
}

8.4. Tweak Existing Version

Route:

PATCH /audio_ad_creator/tweak

Description:

Applies lightweight edits (voice change, music bed swap, SFX adjustments, narration offset).

Request Body:

{
  "audiomessage_id": "ad_123",
  "render_spec": {
    "voice_id": "new_voice",
    "music_bed_id": "synth_pop",
    "narr_offset_ms": 1000,
    "music_gain_db": -2.5,
    "include_chime": false,
    "sfx_events": [
      {"asset_id": "cash_register", "offset_ms": 500, "gain_db": -1}
    ]
  }
}

8.5. Fetch Asset Pickers

Routes:

GET /audio_ad/assets/voices

GET /audio_ad/assets/music_beds

GET /audio_ad/assets/sfx

Description:

Fetches available voices, music beds, and SFX for the picker modals.

8.6. Session State Management

Routes:

GET /audio_ad_creator/session/{audio_message_id}

POST /audio_ad_creator/session/{audion_message_id}

Description:

  • GET: Retrieve full session state (prompt, tokens, tweaks, previews, final output).
  • POST: Save updated session state incrementally.

8.7. Finalization

Route:

POST /audio_ad_creator/save_final

Description:

Marks the selected preview as the final published version.

Saves final_output JSON for production use.

Request Body:

{
  "audiomessage_id": "ad_123",
  "final_output": { ... }
}

8.8. History Management

  • Each generated or tweaked ad version is automatically saved to an history collection.
  • Each version includes:
    • Timestamp
    • Status (draft, saved, published)
    • Preview URL
    • Full render_spec JSON
  • History is fetched as part of session payload; no separate endpoint needed.

8.9. Migration Plan

This section outlines how the transition to the unified AdCreator flow will be safely managed.

Phase 1: Add New Endpoints

  • Implement all new /audio_ad_creator/ routes alongside existing endpoints.
  • Frontend vNext uses only new endpoints; old frontend continues using old ones.

Phase 2: Soft Migration of Audio Messages

  • Extend audio_messages collection with new optional fields:
    • creator_version, prompt_text, tokens, generated_previews, selected_version_id, tweak_form, voice_options, music_options, sfx_library, history, final_output.
  • On first edit/save in vNext, automatically populate these fields for existing ads (without deleting legacy fields).
  • Tag migrated documents with creator_version: vNext.

Phase 3: Gradual Frontend Switchover

  • Enable new UI for:
    • New ads.
    • Existing ads migrated to creator_version: vNext.
  • Fallback to old UI if no creator_version is detected.

Phase 4: (Optional) Background Batch Migration

  • Run a cronjob to fill missing fields in older audio messages.
  • Mark all batch-migrated items with creator_version: legacy_migrated.

Phase 5: Future Cleanup

  • After full adoption (~6 months):
    • Deprecate old routes.
    • Remove unused legacy fields from DB (e.g., options_to_mixed_url).

Notes

  • All new endpoints live under /audio_ad_creator/
  • Full backward compatibility guaranteed.
  • Zero downtime during rollout.
